## Where the Tracks Meet

You've completed either the Testing track, the Observability track, or both. In practice, these two disciplines work best together — testing catches issues before they reach production, and observability catches what testing missed.

This lesson shows how to combine everything into a production-grade workflow.

## The Feedback Loop

The most powerful setup on Cekura is a closed loop:

<Steps>
  <Step title="Test Before Deploy">
    Run evaluators against your agent's latest prompt or configuration. Use Autogen for broad coverage and targeted evaluators for critical flows. Run Infrastructure Suite to validate the voice stack.
  </Step>

  <Step title="Deploy with Confidence">
    Use CI/CD (GitHub Actions) to gate deployments on test results. If critical metrics fail, the deployment blocks.
  </Step>

  <Step title="Monitor in Production">
    Production calls flow into Cekura via the Observe API. Metrics evaluate every call (or a sample). Dashboards track trends. Alerts fire on regressions.
  </Step>

  <Step title="Learn from Production">
    Identify calls where metrics disagree with reality. Send them to Metric Lab. Annotate and Auto Improve.
  </Step>

  <Step title="Generate New Tests from Real Calls">
    Use the **Create Scenarios from Call Logs** feature to turn real production calls into evaluators. This ensures your test suite reflects actual user behavior.
  </Step>

  <Step title="Re-test and Iterate">
    Run the new evaluators against your agent. Use Auto Improve on the prompt. Close the loop.
  </Step>
</Steps>

## Load Testing

Before a major launch, use load testing to validate your agent handles concurrent calls:

1. Select your evaluators
2. Set the **frequency** parameter to run each scenario multiple times (e.g., 5x)
3. Run them simultaneously to simulate peak load
4. Monitor infrastructure metrics (latency, error rate, call drops)

See [Load Testing](/documentation/guides/testing-agents/load-testing) for details.

## Putting It All Together

Here's what a mature Cekura setup looks like:

```
Development:
  ├── Autogen evaluators from agent description
  ├── Manual evaluators for critical flows
  ├── Conditional Actions for complex branching
  ├── Test Profiles + Mock Tools for identity flows
  ├── Red Teaming for security
  └── Infrastructure Suite for voice stack validation

CI/CD:
  ├── GitHub Actions runs tests on every PR
  ├── Cron jobs run smoke tests every 6 hours
  └── Infrastructure Suite runs daily

Production:
  ├── Observe API receives all call data
  ├── Metrics evaluate quality (sampled for cost)
  ├── Dashboards visualize trends
  ├── Alerts notify on regressions
  └── Metric Lab refines metrics from real data

Feedback Loop:
  ├── Production calls → new evaluators
  ├── Failed metrics → Metric Lab → Auto Improve
  └── Test results → Auto Improve → better prompts
```
